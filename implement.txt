                    SYNCSTREAM - YOUTUBE WATCH PARTY SYSTEM
                        INTERN ASSIGNMENT IMPLEMENTATION

This document details the engineering specifications, architectural design, custom 
implementations, and bonus features engineered for the SyncStream real-time 
synchronized YouTube Watch Party application.

--------------------------------------------------------------------------------
1. CORE FEATURES IMPLEMENTED
--------------------------------------------------------------------------------
[✓] Real-Time Playback Synchronization:
    - Action synchronization (Play, Pause, and Seek timestamps) is mirrored instantly 
      across all users in a given room using WebSockets (Socket.IO).
    - Prevents sync drift: Periodically, the Host's active player emits its current 
      playhead timestamp. The server verifies and broadcasts this to align all 
      participants' local players if drift exceeds a 3-second threshold.

[✓] Room-Based Model:
    - Dynamic room generation: Creating a watch party automatically constructs an 
      isolated room with a unique 5-character alphanumeric ID (e.g. "a3x9k").
    - Users can join rooms using the unique Room ID or by visiting a direct shareable 
      share link (e.g. http://domain.com/room/:roomId).

[✓] Embedded YouTube IFrame Player:
    - Fully controlled via the official YouTube IFrame Player API.
    - Synchronized loading of video IDs dynamically across clients.

[✓] WebSockets Real-Time Transport:
    - Managed via Socket.IO for bidirectional, high-performance communication.
    - Built-in reconnection support and dynamic client socket grouping (socket.join).

[✓] Strict Role-Based Access Control (RBAC):
    - Supported Roles: Host (Creator), Moderator, Participant (Default), Viewer.
    - Host Powers: Access to playback controls, promoting participants to Moderators, 
      demoting Moderators, transferring Host ownership, and kicking participants.
    - Moderator Powers: Play, Pause, Seek, and Change Video URL.
    - Participant Powers: Locked playback (watch-only).

2. COMPLETED BONUS FEATURES (ABOVE & BEYOND MVP)
--------------------------------------------------------------------------------
[★] Upstash Redis State Persistence Store:
    - Integrated TLS-secured Upstash Redis (`ioredis`) for state management.
    - All active rooms, playback positions, active video IDs, play/pause states, and 
      member tables survive backend cold starts and Render instance spin-downs.
    - Memory optimization: Idle rooms are garbage collected automatically upon the 
      departure of the last user, or expired via a 24-hour Time-to-Live (TTL) key.

[★] Locked Player Pointer Events:
    - To prevent participants from bypassing restrictions by clicking the iframe player 
      directly, pointer events on the YouTube container are locked:
      `pointerEvents: isAllowedToControl ? "auto" : "none"`
    - Non-control users cannot interact with or click elements in the YouTube player.

[★] Custom Secured Timeline Progress Bar:
    - Hidden YouTube controls: Initialized YT player with `controls: 0` to disable the 
      default bottom bar overlay.
    - Timeline Poller: Created a highly responsive `500ms` polling effect that reads the 
      elapsed time and duration from the YouTube IFrame API.
    - Local Scrubbing: Created an HTML Range slider timeline visible *only* to the Host 
      and Moderators. Scrubbing the timeline triggers synchronized seeks locally and 
      emits coordinates across all active client sockets.

[★] Real-Time Interactive Group Chat:
    - Integrated a real-time chat module inside watch rooms.
    - Messages feature dynamic participant coloring, absolute timestamp logs, and 
      automated container scrolling for optimal user experience.

[★] Real-Time Activity Notification Toasts:
    - Custom slide-in notifications indicate events like joins, departures, role 
      promotions/demotions, kicks, and host transfers.

[★] Dynamic Play Icon Favicon & Brand Identity:
    - Replaced generic placeholders with a highly detailed, professional gradient play 
      button favicon (built with SVG linear gradients and dropping shadows).
    - Updated HTML documentation headers for professional look.

4. WEBSOCKET ARCHITECTURE & ENFORCEMENT FLOW
--------------------------------------------------------------------------------
Below is the data lifecycle for a playback event (e.g. scrubbing the local progress bar):

1. Action: Host/Moderator scrubs the custom timeline slider.
2. Local Event: Local player executes `seekTo(seconds, true)` and fires the socket event:
   `server.emit("sync-time", seconds)`
3. Server Validation:
   - The backend catches `"sync-time"` and loads the room structure from Upstash Redis.
   - Evaluates the sender's socket ID against the user database.
   - Rejects the request if the sender's role is "participant" or "viewer".
   - If valid, updates room playhead in Redis and broadcasts `"sync-time"` to all 
     other active sockets in the target `roomId`.
4. Client Update: Participants catch `"sync-time"` socket event, override state flags, 
   and invoke `playerRef.current.seekTo(time, true)` to match the host instantly.

5. CODE WALKTHROUGH READINESS QUESTIONS
--------------------------------------------------------------------------------
Q: How do WebSockets enable real-time synchronization?
A: Standard HTTP is request-response and cannot push state updates. WebSockets build a 
   persistent, full-duplex TCP tunnel. This enables the server to instantly broadcast 
   events (e.g. pause, play, seek, URL change) to all connected clients in a specific 
   room.

Q: How does the role-based validation work on the backend?
A: Every socket connection contains custom parameters (`socket.role`, `socket.roomId`). 
   When an event is captured, the server queries the database/Redis, matches roles 
   against required permission levels, and strictly halts unauthorized events.

Q: What are the benefits of Upstash Redis over simple in-memory variables?
A: Render's free tier spins down idle instances, losing all in-memory Map variables. 
   Offloading room metadata and active member tables to Upstash Redis ensures 
   reconnecting clients find their watch party exactly where they left it, surviving 
   cold starts.
